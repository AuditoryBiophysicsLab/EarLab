Desktop EarLab: A Virtual Laboratory for Auditory Experimentation

 

System Requirements

  • Pentium III or better
  • Windows 2000 or XP
  • Memory: 256 MB minimum, 512 MB  or more recommended

 

Overview

EarLab is a virtual laboratory that is intended to allow scientists from a wide range of backgrounds to perform experiments on large-scale computational models of the mammalian auditory pathways. The EarLab simulations are based on interchangeable building blocks or modules each of which represents a different component of the auditory system or sound-source, data-acquisition and data-visualization systems. The modules are designed to run in a distributed heterogeneous computing environment and to be configured at run time. The physiological modules are designed to be species independent with species dependent parameters loaded at run time.  The current system includes modules that represent sound sources, propagation between the source and the tympanic membrane, and the subthalamic auditory pathways. These modules can all run on a single CPU (Desktop EarLab) or they can be distributed across many heterogeneous CPUs connected by a local or a wide-area network (Distributed EarLab).

EarLab takes time and divides it up into segments called frames. A frame may continue many one sample or many samples depending on the relative size of the frame and the inter-sample interval.  The modules compute independently during each frame and then exchange information at the end of the frame.

In this document we will use the term model to mean a group of modules and their interconnections.  EarLab experiments consist of multiple runs where a run is a single simulation on a specific model with a single set of parameters. A model consists of a choice of modules and their interconnections. A model diagram file is used to specify the modules to be used, the dimensionality of the module outputs, and the source of module inputs. The module parameters are specified in a parameter file. 

The Desktop EarLab installation package includes the EarLab ExperimentManager, the EarLab Simulator, a library of modules, a library of sound files, data browsing software (EarLab DataViewer), documentation and help files, and a collection of preconfigured models.

Diagram Files

The modules used in a model, their interconnections, and their instance names are specified in a Diagram File.  Diagram files are ASCII files that can be created with any standard text editor. A diagram file is a list of entities, each with one or more records that provide the simulation environment with the information needed to run the simulation. The first entity in a diagram file is the Model entity which is described by a list of the instance names for the modules to be included in the Model. Since a given module may be used more than once in a model, each instance is given a separate name.  A list of available modules and associated documentation can be found at: http://earlab.bu.edu/databases/module/Modules.aspx. An entity in the diagram file is declared by a name in square brackets (e.g. [Model] in the example below). The entity name is followed by one or more lines (records).  Each record contains three fields:

 

Data-type         Parameter         Value

 

The fields can be separated by any combination of spaces and tabs.  In the case of the Model entity, there is only one record. This record contains the list of module instance names as shown below for a simple monaural model:

 

[Model]

// these are the instance names for the modules used in this model

//

string            ModuleList        "DataSource, MiddleEar, NonlinearBM, InnerHaircell, AuditoryNerve, MEsink, BMsink, IHCsink, ANsink"

 

where comment lines are indicated by ”//”.

In this model, “DataSource” is the instance name for a module that will read in a stimulus waveform from a file and send it on to “MiddleEar” which is the instance name for a middle ear module. If we were to create a binaural model, then we would have needed two instances for each of these modules.  The actual executable (DataSource.dll) to be used in the simulation is specified for each instance name as shown below in the first record for DataSource. 

[DataSource]

string            ModuleExecutable  "DataSource.dll"

IntVector         Output.1                [ 100 ]

string            Output.1.DataType "Waveform"

 

The second record for the DataSource entity defines the dimension size for the first output stream for DataSource (this module has only one output stream).  DataSource.dll puts out a one dimensional vector which is the stimulus waveform. The size of the vector indicates how many samples per frame will be in the output stream.

Desktop EarLab supports two types of data, continuous (sampled) waveforms and event times. The DataSource.dll executable puts out waveform data and this is indicated in the third record.

The output from the Data Source entity in this model is the input to the MiddleEar entity.

[MiddleEar]

string            ModuleExecutable        "MiddleEarSimpleFilter.dll"

string            Input.1                 "DataSource.1"

IntVector         Output.1                [ 100 ]

String            Output.1.DataType       "Waveform"

 

The first record for the MiddleEar entity indicates that the executable will be MiddleEarSimpleFilter.dll and the second record indicates that the first input for the MiddleEar entity will be DataSource.1, the first output of the DataSource entity.

The output of DataSource is the input to the NonlinearBM entity which simulates basilar membrane mechanics.  In this case, the output of NonlinearBM is a 2-dimensional matrix. The first dimension is the number of samples per frame (100) and the second dimension corresponds to the number of locations (64) along the basilar membrane that are being simulated.

[NonlinearBM]

string            ModuleExecutable  "BasilarMembraneNonlinearGammatone.dll"

string            Input.1           "MiddleEarSimpleFilter.1"

IntVector         Output.1          [ 100 64 ]

string            Output.1.DataType "Waveform"

 

This example model also includes a number of entities whose role is to store output data from the simulation modules (MEsink, BMsink, IHCsink, ANsink).  These are all instances of the executable "DataSink.dll".

 

Parameter Files

The parameter file provided modules with information about how they are to perform their functions. Through these parameters, it is possible to change input and output file names at run time as well as change parameters that change basilar membrane frequency-place maps and middle-ear frequency response.  As was the case for the Diagram File, Parameter Files contain sections for each entity in the Model and each section consists of a list of records.  For example, the parameters for DataSource include the sampling rate in Hertz, the name of the input file, and a flag to indicate that the input file is a Wave-format file.  The DataSource DLL can read both Wave and ASCII files. The output of DataSource for sound files will be in units of μPa.  For the special case of Wave files, DataSource normalizes the stored data by the maximum possible value (32,768 for 16-bit files) and then multiplies by the WaveFileScaleFactor.  Because Wave files can be stereo, the WaveFileChannelNumber allows the user to use channel-0 (left) or channel-1 (right).  DataSource also is capable of adding a fixed delay to the stored waveform.  This delay is an integer number of samples specified by OutputDelay_Samples. To get the delay in units of seconds, multiply OutputDelay_Samples by SampleRate_Hz.

[DataSource]

float       SampleRate_Hz           100000

string      InputFileName           "C:\Earlab\Signals\Tones\1000_Hz.wav"

bool        InputIsWaveFile         true

int         WaveFileChannelNumber   0

float       WaveFileScaleFactor     2000

int         OutputDelay_Samples     0

 

Since there is considerable variation in the characteristics of auditory structures across species, many modules expose parameters that can be changed in order to simulate different species.  For example, the simple middle ear module used in this example, uses a high-pass filter to simulate the middle-ear transfer function.  In our example, the parameter HighpassCornerFreq_Hz controls the corner frequency of the filter and we have set it to 500 Hz in order to simulate the human middle ear. The other filter parameters are the filter order (HighpassFilterOrder) and the pass-band gain (Gain).

[MiddleEar]

// These are approximate human parameters

float       SampleRate_Hz                 100000

float       HighpassCornerFreq_Hz         500

int         HighpassFilterOrder           2

float       Gain                          30

[MEsink]

float       SampleRate_Hz           100000

bool        ProduceOutputFile       true

bool        BinaryOutput            true

string      OutputFileName          "MiddleEar.1.binary"
